# CI Review — 2026-05-29

> **Audience:** operator + future wizard sessions. Read this once per quarter to recalibrate; check the headline metric weekly.
>
> **Method:** 14-day window (2026-05-15 → 2026-05-29) — ambient.jsonl failure events + merged PR titles + observed operator/agent friction across this session and the prior week.
>
> **Verdict:** the CI surface is functional but expensive to keep running. The bottleneck is reliability, not speed. Five concrete levers below; one of them (containerize on Linux) cuts the rot tax in half.

## What we have NOT seen hold back CI

These are the optimization-where-it-doesn't-matter traps. Avoid spending here:

- **Raw cargo build speed.** Wave 1 (sccache+R2 + nextest) addressed this. After cache warming, builds are fast enough. Marginal returns from further compilation optimization.
- **Network / GitHub API capacity.** GraphQL exhaustion (132 events) is real but cache absorbs most; restructured calls already in place.
- **Operator's code-writing throughput.** When CI works, ships flow fine. The bottleneck is what happens when CI *doesn't* work.
- **Test suite size.** ~969 `scripts/ci/*.sh` test scripts is a lot but they're cheap to run individually; the cost is in serial gating not test count.

## Top-5 highest-leverage levers

Ranked by `time-saved / effort × confidence`. Each lever cites the failure classes it eliminates.

### Lever 1 — Containerize CI (Docker on Linux substrate)

**Eliminates:** Runner-state corruption (rank 1, ~6–8 h/14d), partial-cascade effects (rank 5)

**Effort:** s for proof-of-concept (GH-hosted ubuntu containers for non-macOS jobs); m for self-hosted Docker on Linux box (~$800 Beelink SER9 per `CI_SCALING_REFERENCE.md` Option C)

**Expected reduction:** ~50% of CI-rot tax (the entire rank-1 class becomes impossible by construction; jobs start from a deterministic image every time)

**Caveats:** Docker on **Mac** is the wrong move — VM overhead gives back what Wave 1 just won. Docker on **Linux** is fast and clean. macOS-only jobs (desktop Tauri build, ACP smoke test) stay on the Mac runners.

**Concrete steps:**
1. Move `cargo-test`, `cargo-fmt`, `clippy`, `audit` to GH-hosted ubuntu (no hardware change required) — drops the corruption class for ~70% of CI work overnight
2. Keep ACP smoke, desktop build, e2e-pwa on Mac (genuinely macOS-dependent)
3. (Optional Wave 2) Provision a Linux box for self-hosted Docker if GH-hosted runtime becomes a bottleneck

---

### Lever 2 — Per-PR isolation of mutable runner state

**Eliminates:** Cascade effects (rank 5), partial CI-config drift (rank 2 — registry/src races)

**Effort:** s — change CI workflow to use unique `CARGO_TARGET_DIR` per job ID + per-PR registry/src directory

**Expected reduction:** ~15% of CI-rot tax (one PR's corruption no longer poisons the next)

**Caveats:** Trades cache hit rate for isolation. sccache+R2 keeps the compile-cache layer; only target/ and registry/src/ go ephemeral. Worth the trade given how many trunk-REDs were "one PR poisoned the runner, next PR failed."

---

### Lever 3 — Make bot-merge.sh event-driven (kill the polling)

**Eliminates:** bot-merge stalls (rank 4, ~2 h/14d), implicit chain of manual `Bot-Merge-Bypass` corrective work

**Effort:** m — rewrite bot-merge.sh's CI-wait loop to use GitHub webhooks (or the GH cache layer the fleet already uses) instead of `sleep 30; gh pr view ...` polling

**Expected reduction:** ~10% of CI-rot tax + eliminates the operator-confusion class ("did bot-merge hang or is it still working?")

**Caveats:** Webhook receiver already exists (`scripts/ops/github-webhook-receiver.py` per the cache-first reads pattern in CLAUDE.md). This lever just wires bot-merge to consume from it.

---

### Lever 4 — Move CI-parity gate from main to PR-time

**Eliminates:** CI-config drift catches (rank 2, ~4–5 h/14d, the most-repeated PR-level failure class)

**Effort:** xs — `test-preflight-ci-parity.sh` already exists; just promote it from advisory to required check, OR fire it in a pre-commit hook

**Expected reduction:** ~15% of CI-rot tax (every "you added a step without allowlisting it" becomes caught locally before push, not on PR's third CI cycle)

**Caveats:** Requires updating the existing `preflight-ci-parity-exceptions.txt` allowlist surface every time a legit new step lands. That's already the operational reality; making it strict just enforces it before main pollutes.

---

### Lever 5 — Auto-flip-to-done on PR merge (gap registry drift)

**Eliminates:** Operator-attention sinks (rank 6 — gap-registry drift class). Also: the dispatch chain bug pattern where META-090/META-122 land but agents/operator can't see the registry-side proof.

**Effort:** xs — add a GitHub Action that runs `chump gap ship <ID> --closed-pr <#>` on PR merge, matching gap-ID from the PR title (regex: `^(?:[a-z]+\()?(INFRA|EFFECTIVE|RESILIENT|DOC|META|MISSION|CREDIBLE)-\d+`)

**Expected reduction:** ~5% of CI-rot tax + cleans up the 35+ silent-shipped gaps clogging the "untraced P0" digest

**Caveats:** Idempotent — re-running `chump gap ship` on an already-done gap is no-op per the existing CLI. Safe to add as a post-merge hook.

---

## Combined impact estimate

Implementing all 5 levers in order: ~50% reduction in CI-rot tax (from 37% → ~18% of ship throughput). That recovers ~9 hours of operator+agent time per 14-day window. Equivalent to ~10 additional product-facing ships per cycle.

Levers 1, 2, 4 are deterministic. Levers 3, 5 are best-effort. Lever 1 has the biggest single impact but the longest tail (hardware decision required if Linux box path).
