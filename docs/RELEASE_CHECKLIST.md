# Release Checklist

Chump uses [cargo-dist](https://axodotdev.github.io/cargo-dist/) for release automation. Cutting a release is just `git tag && git push`; everything else is automatic.

## Prerequisites (one-time, already done)

- [x] `cargo-dist` configured via `dist-workspace.toml`
- [x] `.github/workflows/release.yml` generated by `dist generate`
- [x] `Cargo.toml` has `repository`, `homepage`, `license` fields
- [x] Workspace members excluded from dist via `[package.metadata.dist] dist = false`
- [x] `CHANGELOG.md` follows Keep a Changelog format
- [x] **Repo-level workflow permissions set to `write`.** The release workflow declares `contents: write` in its `permissions:` block, but GitHub also enforces a repo-level default that can override it. If that default is `read`, the "Create GitHub Release" step 403s on `POST /repos/.../releases` even though the platform-build artifacts succeed. Set once with:
  ```bash
  gh api --method PUT "repos/{owner}/{repo}/actions/permissions/workflow" \
    -f default_workflow_permissions=write \
    -F can_approve_pull_request_reviews=false
  ```
  Verify with `gh api "repos/{owner}/{repo}/actions/permissions/workflow"`. If a release fails with `HTTP 403: Resource not accessible by integration` after the build matrix succeeds, this is almost certainly the cause. Re-running the failed jobs won't help because reruns inherit the original workflow run's permission snapshot — you must delete and re-push the tag to trigger a fresh run that picks up the new permissions.

## Per-release steps

### 1. Update version

Bump version in `Cargo.toml`:

```toml
[package]
version = "X.Y.Z"
```

Follow SemVer:
- **Patch** (0.1.0 → 0.1.1): bug fixes, doc changes, no API changes
- **Minor** (0.1.0 → 0.2.0): new features, backward compatible
- **Major** (0.1.0 → 1.0.0): breaking changes

### 2. Update CHANGELOG.md

Move items from `[Unreleased]` into a new versioned section:

```markdown
## [Unreleased]

## [X.Y.Z] — YYYY-MM-DD — Short description

### Added
- ...
### Changed
- ...
### Fixed
- ...

[X.Y.Z]: https://github.com/repairman29/chump/releases/tag/vX.Y.Z
```

### 3. Verify the plan

```bash
cargo check                        # compile clean
cargo test                         # all 530+ tests pass
cargo clippy --all-targets -- -D warnings
dist plan                          # check what will be released
```

`dist plan` should show:
- `announcing vX.Y.Z`
- `rust-agent X.Y.Z` with 4 archives (aarch64/x86_64 × apple-darwin/linux-gnu)
- `rust-agent-installer.sh` + `sha256.sum`
- No chump-desktop, chump-mcp-*, etc. (excluded via metadata)

### 4. Commit

```bash
git add Cargo.toml Cargo.lock CHANGELOG.md
git commit -m "chore(release): vX.Y.Z"
git push
```

### 5. Tag and push

```bash
git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z
```

### 6. Wait for CI

The tag push triggers `.github/workflows/release.yml`. It will:

1. **Plan** (Ubuntu) — parse config, decide what to build
2. **Build local artifacts** (matrix: macOS arm64, macOS x86_64, Linux x86_64, Linux aarch64) — produces `.tar.xz` archives + checksums
3. **Build global artifacts** (Ubuntu) — produces `rust-agent-installer.sh` and `sha256.sum`
4. **Host** — create a GitHub Release, upload all artifacts, write release notes from CHANGELOG.md
5. **Announce** — finalize release

Monitor at: https://github.com/repairman29/chump/actions

Typical duration: **15-25 minutes** (Linux aarch64 cross-compile is the slowest).

### 7. Verify the release

Once CI succeeds:

```bash
# Check GitHub Release exists
gh release view vX.Y.Z

# Verify installer works
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/repairman29/chump/releases/download/vX.Y.Z/rust-agent-installer.sh | sh

# Verify installed binary runs
chump --check-config
```

### 8. Update ACP Registry entry (if needed)

If this is a material release (features, auth changes), bump the version in `registry-submission/chump/agent.json` and PR it to `agentclientprotocol/registry`.

For patch releases, the registry auto-detects the new GitHub release within an hour — no action needed.

## Troubleshooting

### Build fails on Linux aarch64

cargo-dist uses [`cross`](https://github.com/cross-rs/cross) for Linux cross-compiles. If ring, openssl, or another native dep fails to build, consider adding to `dist-workspace.toml`:

```toml
[dist]
github-build-setup = "../build-setup.yml"
```

...with custom `apt-get install` steps.

### Installer URL 404s

Check that `dist-workspace.toml` lists `installers = ["shell"]`. If it does, the installer is at:

```
https://github.com/repairman29/chump/releases/download/vX.Y.Z/rust-agent-installer.sh
```

(archive name is `rust-agent-*` because the crate package name is `rust-agent` — the binary inside is `chump`.)

### Need to re-release after a botched tag

```bash
git tag -d vX.Y.Z                  # local
git push origin :refs/tags/vX.Y.Z  # remote
# fix issue, commit, then re-tag
```

Then delete the GitHub Release manually via the UI (`gh release delete vX.Y.Z`).

## Useful commands

```bash
dist init                # reconfigure
dist generate            # regenerate .github/workflows/release.yml
dist plan                # preview what would be released
dist build               # build locally (without publishing)
dist manifest            # show full manifest
```

## Why cargo-dist?

- Purpose-built for Rust binary releases (same folks as [Axo](https://axo.dev))
- Handles cross-compile matrix (macOS universal via `lipo`, Linux via `cross`)
- Generates checksums, SLSA provenance (optional), release notes
- Writes a reliable `install.sh` — `curl | sh` for users, like Hermes's pattern
- Used by hundreds of Rust OSS projects — stable, well-documented
- We evaluated: manual Actions matrix (tedious), rust-action/release (less polished), [cargo-release](https://github.com/crate-ci/cargo-release) (for crates.io, not binaries). cargo-dist fits the binary-release use case best.
