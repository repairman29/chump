## 2. Hypothesis

**Primary hypothesis (H1) — falsifiable.**
> If the spawn-time lessons block is enabled (`CHUMP_LESSONS_AT_SPAWN_N=5`,
> top-5 recency × frequency-ranked rows from `chump_improvement_targets`
> prepended to the assembled system prompt), then **task success rate**
> on the gap-shaped task population (defined in §3) will change by
> **|Δ| ≥ 0.05** in **either direction** relative to the
> `CHUMP_LESSONS_AT_SPAWN_N=0` control, with Wilson 95% CI excluding zero.

The two-sided form is intentional: COG-024 framed lessons-OFF as the
safe default precisely because injection has been shown to *help* small
models on reflection fixtures (EVAL-025) and *harm* sonnet-class models
on the same fixture (EVAL-027c). This gap measures the effect on the
*gap-shipping task population* — different task class, different
cognitive demand — and either direction is a publishable result.

**Null hypothesis (H0).**
> No detectable effect on task success rate: |Δ(B − A)| ≤ 0.05, or
> Wilson 95% CI on the delta overlaps zero. Lessons injection neither
> helps nor hurts on this substrate / task population.

**Alternative explanations to rule out:**
- *Spawn-time prompt-length confound* — the lessons block adds ~500–1500
  tokens of prefix; an apparent Cell-B effect could be context-window
  pressure rather than lesson content. **Addressed by Cell C** (N=10),
  which doubles the lesson-block size: if the C effect tracks the B
  effect monotonically, the mechanism is content; if C diverges
  non-monotonically (especially worse than B), context pressure is the
  more likely mechanism.
- *Lesson-set staleness* — the top-5 ranking depends on
  `chump_improvement_targets` content at sweep time. **Addressed** by
  snapshotting that table to `logs/ab/COG-032/lessons_snapshot.sql` at
  sweep launch and locking it across all cells; same lesson set seen by
  every Cell B / C trial regardless of when it runs.
- *Judge-family bias* — Anthropic judges have shown a systematic
  reward-hallucinated-tool-calls pattern (EVAL-010). **Addressed** by
  the cross-family judge panel in §3.
- *Task-distribution drift between cells* — different gaps land in
  Cells A vs B by chance. **Addressed** by deterministic round-robin
  task assignment from a frozen task list (§3) — every cell sees the
  same 50 task IDs in the same order.

## 3. Design

### Cells

| Cell | Intervention | Expected direction |
|---|---|---|
| A (control) | `CHUMP_LESSONS_AT_SPAWN_N=0` (no spawn-time lessons block) | neutral baseline |
| B (intervention) | `CHUMP_LESSONS_AT_SPAWN_N=5` (top-5 recency×frequency lessons prepended) | unknown — H1 predicts \|Δ\|≥0.05 in either direction |
| C (sensitivity) | `CHUMP_LESSONS_AT_SPAWN_N=10` | tests dose-response: monotonic vs non-monotonic |

Cell C is included rather than dropped on budget grounds: without it,
any Cell-B effect is confounded with prompt-length pressure and the
result cannot inform a default-flip decision. Cell C is only
~$8 incremental on the curated bench (§ Substrate) and is the cheapest
mechanism check we can buy.

### Task population

**50 tasks per cell, drawn from a frozen curated bench set**:
`scripts/ab-harness/fixtures/cog032_gap_bench_v1.json` (to be authored
before sweep launch, locked at preregistration commit time).

**Choice rationale.** The acceptance criteria in `docs/gaps/COG-032.yaml`
say "execute 50 gaps in each condition" which sounds like real registry
gaps. We use a **curated bench** instead because:

1. **Repeatability.** Real `INFRA-*` gaps mutate `.chump/state.db`,
   ship PRs, and disappear from the open pool — Cell B literally cannot
   re-execute the same gap Cell A already shipped. A curated bench is
   replayable across all three cells, which is what makes paired
   analysis (§7) possible.
2. **Confounder control.** Real-gap difficulty is non-stationary
   (the easy ones get picked first, the hard residue accumulates). Over
   a 3-cell sweep this would systematically advantage whichever cell
   ran first. The bench freezes difficulty at preregistration time.
3. **Substrate cost.** Real-gap execution involves fleet coordination,
   merge-queue interaction, and cross-session lease contention — none
   of which we want as variance in the lessons-injection signal.

**Realism caveat.** A curated bench underrepresents the messiness of
real-gap shipping (rebase conflicts, stale binaries, sibling-session
stomps). This is an explicit scope limitation; the result generalizes
to "lessons injection on bench tasks of this difficulty distribution,"
not "lessons injection in the production fleet." A follow-up gap
(filed at result time if H1 supported) would re-run on real gaps.

**Bench composition (locked):**
- 25 INFRA-class tasks (CLI / scripting / git-hooks / SQLite — historical
  median fleet difficulty)
- 15 cognitive / faculty tasks (src/{system_prompt,reflection_db,
  briefing,agent_loop}.rs — directly relevant to the lessons mechanism)
- 10 doc / process tasks (mechanical, low-cognitive — included to test
  whether lessons help on tasks where they obviously shouldn't fire,
  i.e. trivial-token contamination per EVAL-030)

Task IDs and reference solutions live in
`scripts/ab-harness/fixtures/cog032_gap_bench_v1.json`. Same 50 task IDs
in same order across all three cells (deterministic).

### Substrate

**Primary substrate:** `claude-opus-4-7` via Anthropic API.

Rationale: this is the model the Chump fleet actually dispatches with
(see CLAUDE.md § Dispatched-subagent backend default = `claude`). The
result must apply to that substrate to inform the COG-024 default flip.

**Recommended secondary substrate (separate cell pair, run if budget
permits and primary signal is ambiguous):** `qwen3:14b` on a 24GB
Apple-silicon Mac via the chump-local backend
(`CHUMP_DISPATCH_BACKEND=chump-local`, `OPENAI_API_BASE` pointed at the
local Ollama instance). This validates the offline-solo-dev mission
(MEMORY: `project_offline_local_llm_mission.md`) and exposes any
substrate-tier interaction (per RESEARCH_INTEGRITY.md, lessons
injection has shown opposite signs on haiku vs sonnet — qwen3:14b sits
roughly at the haiku tier and may show a different sign than opus).

**Hardware.** Cloud-Anthropic for Cell pair (A, B, C) on opus-4-7. If
the qwen3:14b pair runs, it executes on the same M4-class 24GB Mac to
match the dogfood substrate.

### Randomization & order

- **Trial order within a cell:** deterministic by task ID
  (alphabetical) — no per-cell shuffling. Variance from order effects
  is shared across cells, so it cancels in the paired delta.
- **Cell execution order:** A, B, C interleaved per task — for each
  task ID, run A, then B, then C, then move to next task. This
  averages slow-API-day effects across cells. Logged to
  `logs/ab/COG-032/run.jsonl` with timestamp + cell tag.
- **Seed discipline:** agent invocation is `claude -p --dangerously-skip-permissions`
  (Claude Code SDK has no exposed seed knob). Substrate non-determinism
  is the dominant noise term; the A/A baseline (§ A/A) measures it.

## 4. Primary metric

**Binary task success rate per cell.**

A trial counts as **SUCCESS** iff *all* of:
1. Agent ships a PR via `bot-merge.sh` within **N = 90 minutes** of
   wall-clock from agent invocation, AND
2. The PR's required CI checks (`test`, `audit`, `ACP smoke test`) all
   reach success state on the agent's branch within an additional 30
   minutes (auto-merge timeout window), AND
3. The PR's diff contains a non-empty change to at least one file
   referenced by the bench task's reference solution paths.

Otherwise the trial is **FAILURE**. Timeouts, agent crashes, refusals,
empty PRs, CI failures, and gap-claim collisions all count as failures
(they are real outcomes of the spawn-time prompt change in production).

```python
def trial_success(trial) -> bool:
    if trial["pr_number"] is None: return False
    if trial["wall_clock_minutes"] > 90: return False
    if not trial["ci_green"]: return False
    if trial["ci_green_minutes"] > 120: return False  # 90 + 30
    if not any(p in trial["pr_diff_paths"]
               for p in trial["bench_task"]["reference_solution_paths"]):
        return False
    return True
```

**Reporting format:** point estimate + Wilson 95% CI per cell;
B−A and C−A pairwise deltas with bootstrap 95% CIs (10000 resamples,
paired by task ID); A/A noise floor cited inline.

## 5. Secondary metrics

Four, each defined explicitly:

1. **Revision count** — number of commits the agent pushed before
   `bot-merge.sh` armed auto-merge (read from `gh pr view --json commits`).
   Captures whether lessons reduce false-start churn.
2. **Time-to-ship (minutes)** — wall-clock from agent invocation to
   `bot-merge.sh` exit-success. Conditional on success only (failed
   trials excluded from this metric).
3. **Pre-commit guard violations** — count of pre-commit hook failures
   in the agent's session (read from agent transcript; pattern-match
   `\[(.*?)\] (FAIL|BLOCK)`). Captures whether lessons reduce
   recurring-violation classes (TBD: this metric directly tests the
   "lessons help" mechanism — if it doesn't move, lessons aren't being
   used even if primary moves).
4. **Inter-judge agreement (κ)** on the primary metric — cross-family
   binary agreement between the two judge families on every trial's
   SUCCESS/FAILURE label. Reported per cell.

**Single-judge waiver, secondary metrics 1–3:** revision count,
time-to-ship, and pre-commit violation count are *mechanically
measured* (read from `gh`, transcripts, hook output) — there is no
LLM-judge call involved. Cross-judge audit does not apply and is
**single_judge_waived: true (mechanical measurement)** for these three.

Metric 4 (κ) is itself the cross-judge audit instrument and is by
definition multi-judge.

**Planned n:** 50 per cell × 3 cells = 150 trials primary substrate.

## 7. Analysis plan

**Primary analysis (preregistered):**

1. Compute success rate per cell (A, B, C) with Wilson 95% CIs on n=50.
2. Compute paired deltas B−A and C−A — paired by task ID — with
   bootstrap 95% CIs (10000 resamples).
3. Test H1 by checking whether the B−A 95% CI excludes zero. Direction
   not preregistered (two-sided test).
4. Compare delta to the A/A noise floor (§ A/A baseline). Delta
   magnitude must exceed **3 × A/A standard deviation** to be
   interpretable.
5. Examine C−A: dose-response shape.
   - **Monotonic same-sign (C−A and B−A both positive or both
     negative, |C−A| ≥ |B−A|):** evidence consistent with content
     mechanism.
   - **Non-monotonic (C−A opposite sign of B−A, or |C−A| materially
     smaller than |B−A|):** evidence consistent with context-pressure
     mechanism — lessons help up to a dose then start to harm.
   - **Both deltas null:** consistent with H0; no mechanism call.

**Secondary analyses (preregistered):**
- Subgroup analysis by bench-task class (INFRA / cognitive / doc).
  Lessons are *expected* to fire harder on the cognitive subset
  (mechanism: lesson content overlaps faculty code more than mechanical
  CLI work), and *not at all* on the doc subset (per EVAL-030
  task-class-aware gating, doc edits are below the trivial-token
  threshold). If the subgroup pattern matches this prediction, it is
  evidence the spawn-time injection respects the existing gating.
- Per-judge κ on the primary SUCCESS label.
- Time-to-ship delta and revision-count delta per cell.

**Mechanism-analysis trigger (per RESEARCH_INTEGRITY.md §4):**
**any** |delta| > ±0.05 on the primary metric requires a written
mechanism analysis. Specifically:
- Sample 5 winning Cell-B trials and 5 losing Cell-B trials.
- For each: extract the assembled prompt's lessons block and the
  agent transcript.
- Classify by hand: did the agent (a) reference a specific lesson,
  (b) avoid a behavior the lesson warned against without naming it,
  (c) ignore the lessons block entirely, (d) violate a lesson?
- Report classification distribution. If ≥ 60% of Cell-B winners are
  category (c), the apparent benefit is unrelated to lesson content
  (likely prompt-length / framing artifact).
- **Cross-judge κ on the mechanism class labels** must hit ≥ 0.60
  (RESEARCH_INTEGRITY.md EVAL-093 gate) before any "lessons fired"
  claim is published.
- **Evaluation-awareness candidate (per RESEARCH_INTEGRITY.md
  EVAL-087, 2026-05-02):** the bench is harness-style — task IDs,
  reference solution paths, structured prompts. Per the integrity
  directive, mechanism analysis must explicitly consider whether the
  observed delta is evaluation-context-detection rather than lesson
  effect. Cite RESEARCH-026 / EVAL-094 result on the comparable fixture
  class, or — if EVAL-094 has not yet shipped on a sufficiently similar
  fixture — note this as an explicit unresolved confound in the result
  doc.

**Exploratory analyses (allowed but clearly labeled):**
- Per-lesson firing rate: which of the top-5 lessons appear most often
  in winning Cell-B prompts? Does any single lesson dominate the
  effect?
- Tier interaction (only if the qwen3:14b secondary pair runs):
  is the sign of the B−A delta the same on opus-4-7 and qwen3:14b?

## 8. Exclusion rules (lock before data collection)

A trial is **excluded** from analysis iff:
- Anthropic API returned 5xx after 3 retries (instrument failure, not
  outcome).
- The bench harness itself crashed before the agent emitted any output
  (config / fixture failure).
- The lessons-block snapshot at sweep launch was empty (`SELECT count(*)
  FROM chump_improvement_targets` returned 0) — invalidates the
  intervention; reschedule.
- `CHUMP_LESSONS_AT_SPAWN_N` was not honored at agent spawn time
  (verify by grep'ing the assembled prompt log for the lessons block
  marker; absence in Cell B/C or presence in Cell A means the env-var
  plumbing failed for that trial).

All exclusions logged with reason. **Exclusion rate > 10% per cell
invalidates the sweep**; we re-run after fixing the cause.

Notably **NOT excluded** (these are real outcomes):
- Agent timeouts (count as FAILURE).
- Pre-commit guard hits (count as FAILURE if they prevent ship).
- Agent refusals (count as FAILURE).
- CI failures unrelated to the agent's diff (count as FAILURE — the
  agent could have noticed and rebased).

## 9. Decision rule

**If H1 supported, B−A direction positive (|Δ| ≥ 0.05, CI excludes 0,
B success > A success):** file a follow-up gap to flip the COG-024
default — `CHUMP_LESSONS_AT_SPAWN_N=5` becomes the new default for
opus-4-7 dispatch. Document the result + mechanism analysis in
`docs/eval/COG-032-lessons-feedback.md`. Cite the magnitude with
the standing RESEARCH_INTEGRITY.md EVAL-087 caveat — "this magnitude
may be inflated or deflated by evaluation-context confounding until
EVAL-094 ships on a comparable fixture."

**If H1 supported, B−A direction negative (B success < A success):**
COG-024's safety-default was correct on this substrate; close COG-032
with that result + mechanism analysis. File a follow-up to
*hard-disable* spawn-time injection on this substrate, and re-run on
qwen3:14b before generalizing.

**If H0 (|Δ| ≤ 0.05 or CI overlaps zero):** COG-024's default is
neither helping nor hurting the *primary* metric. Examine secondary
metrics: if revision count or time-to-ship moved meaningfully, the
result is "lessons change *how* the agent works without changing
*whether* it ships," which still informs the default decision.
Write up as a null/secondary-effect result in
`docs/eval/COG-032-lessons-feedback.md`.

**If ambiguous (CI wide, mean directional but overlapping zero):**
escalate to n=100 per cell on a follow-up gap; do not flip COG-024
default on the n=50 result.

## 11. Falsifying condition

The hypothesis "spawn-time lessons injection helps gap-shipping
agents on the opus-4-7 substrate" is **falsified** by the observation:

> **Cell B success rate ≤ Cell A success rate within Wilson 95% CI on
> the primary metric (n=50 per cell), AND |Cell B − Cell A| ≤ 0.05.**

Equivalently: if H0 holds, or if H1 holds in the negative direction,
the "lessons help" hypothesis is falsified on this substrate / task
population. The result is *publishable as-is* — null and negative
results inform the default-flip decision and ship to
`docs/eval/COG-032-lessons-feedback.md` either way.

## 12. A/A baseline

**Required by RESEARCH_INTEGRITY.md § 5 ("Every eval series must
include at least one A/A run").**

**Protocol.** Before Cell B / C launch, run the **first 10 task IDs of
the bench** through Cell A *twice* — same env, same agent invocation,
same task list. Call these A1 and A2.

**A/A noise floor metric:** σ = standard deviation of (A1[i] −
A2[i]) over the 10 paired trials. Wilson CI on the |A1 − A2| mean
delta.

**A/A pass criterion:** **|mean(A1 − A2)| ≤ 0.05 AND σ ≤ 0.10**. If
|mean| > 0.05 or σ > 0.10, **the experiment is invalidated** —
substrate noise is too high to detect a 0.05 effect in the main
sweep, and we must redesign (likely n ≥ 100 per cell or a more
deterministic agent loop) before continuing.

**Citation in result doc:** A/A σ value cited inline next to every
B−A and C−A delta, formatted as `Δ = X (±CI), σ_AA = Y`. Any cited
delta < 3 × σ_AA is labeled "within A/A noise."
